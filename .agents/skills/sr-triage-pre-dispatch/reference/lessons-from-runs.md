# Lessons from runs (the empirical foundation)

> **Purpose.** This file captures the empirical patterns the triage skill is built on. Two real runs in April 2026 produced the lessons. As new runs surface new patterns, append to this file first; promote into a new check or decoder rule only after the pattern repeats.

## Run A — Plumbing batch (April 2026)

**Scope:** 259 unique plumbing SRs from a 2-week window across Ulta (118 SRs), Luxottica (17), DriveTime, Boyd Group, Circle K, Lenscrafters, and others. Single-trade focus on plumbing.

**Intent:** One-time learning batch. Designed to surface what an agent could see across a high volume of like-trade SRs in a short window. Not a normal operating cadence.

**Method:** Augmented CSV with per-SR scoring (right / wrong / partial / insufficient on multiple flags), Cursor Canvas dashboard, customer-issue-brief delivered to Carrie Kroeker (Ulta), Aaron Dixon (Luxottica), Amy Chasse (DriveTime).

**Source:** `docs/artifacts/sr-quality-review/customer-issue-brief.md`, `docs/artifacts/sr-quality-review/Plumbing_revised_assessed_latest.csv`, `docs/artifacts/sr-agent-runs-technical-review-derek-2026-04-30.md`.

### Patterns surfaced

| Pattern | Evidence | Promoted to |
|---|---|---|
| Agent reads SRs in isolation; misses parent/child SR linkage. | Ulta Lincoln 0535 (SR SR-EXAMPLE-002): agent reported "no replacement quote in flight" while child SR SR-EXAMPLE-003 was already open with the replacement scope. | `sr-graph-resolver.md` (Rule 1) |
| Agent mistranslates Siebel sub-statuses. | Ulta Riverbank 0784 (SR SR-EXAMPLE-001): agent translated `Known ETA` as "closed to bill prematurely." Vendor was entering invoice line items while awaiting Ulta corporate direction. | `sub-status-decoder.md` (full dictionary) |
| Agent attributes data-migration gaps to Tech Dispatch. | Ulta SLX → Siebel asset migration: stores rarely surface data tags. Ten-plus Ulta SRs flagged with "missing asset data" should have been classified as migration / asset master, not as Tech Dispatch failure. | `asset-data-lineage.md` |
| Recurring-failure pattern is the highest-leverage pre-dispatch trigger. | Ulta Rockford 0370 (SR SR-EXAMPLE-005): tankless replaced August 2025, on its 4th visit (priors: SR-EXAMPLE-006 breaker reset, SR-EXAMPLE-007 power cycle, SR-EXAMPLE-008 E003 ignition / vinegar descale, SR-EXAMPLE-009 supply lines). The fourth flame-sensor cleaning was treated as a routine repair when it should have been a customer call recommending replacement. | `pre-dispatch-checks.md` (Check 4 customer call recommendation) |
| Intake miscategorization is a productionizable rule. | Ulta Minnetonka 0065 (SR SR-EXAMPLE-010): symptom logged as "leaking pipe," actually a drain backup. First SP was a generalist plumber, not a drain specialist. Field surfaced the actual need. | Future intake-validator primitive (FacilitiesOS backlog) |
| NEC loaded coverage data is systemically wrong. | Ulta Capitola 0743 (SR SR-EXAMPLE-011): NEC declined as out of area despite loaded coverage. Maria + Patros have rebuilt NEC coverage three times since the transition. Account-wide issue, not site-specific. | `MAGIC/memory/actions.md` (NEC coverage operational action) |

### Customer feedback that drove this skill

Carrie Kroeker reply on 2026-04-30: 2 of 5 reviewed SRs were clean agent hits (Minnetonka intake, Rockford recurring asset). 3 of 5 were partial — agent flagged the right concern but missed live operational context. She named three system-level signals to fix: parent/child linkage, sub-status decoding, asset-data lineage. All three are now decoders in this skill.

Source: `docs/artifacts/sr-quality-review/carrie-ulta-response-synthesis-2026-04-30.md`.

## Run B — Cold and frozen beverage cohort (April 2026)

**Scope:** 11 SRs across Circle K (7) and Refuel (4) in a 2-week window. Multi-trade (cold beverage equipment, ice machines, fountain beverage, combo units).

**Intent:** Hand-curated cohort review. Designed to go deep on a small set with full SR detail, notes, time events, invoices, line items, asset records.

**Method:** Per-SR review via Gateway and VixxoLink, scored against the cold-beverage trade skill, cross-SR pattern aggregation, action-item capture.

**Source:** `docs/artifacts/cold-frozen-beverage-sr-review-final-2026-04-29.md`, `docs/artifacts/cold-beverage-cohort-retrospective-2026-04-29.md`.

### Patterns surfaced

| Pattern | Evidence | Promoted to |
|---|---|---|
| LOS + Short Description does not map to the governing trade skill. | 9 of 11 SRs booked under "Cold Beverage Equipment" but needed a different governing skill (ice machines, fountain beverage, or combo unit). | `pre-dispatch-checks.md` (Check 5 trade skill routing). Mapping itself is a separate FacilitiesOS backlog item. |
| SP write-up discipline varies by SP; root cause is upstream onboarding compliance. | Air America Refrigeration: zero SP-side write-up across 3 SRs in 5 days. Brought on without standard onboarding training; never corrected operationally. | `post-facto-application.md` (Check 6 documentation quality scoring). SP onboarding compliance audit is a separate operational action. |
| Replace-don't-repair pattern at SP level. | Crete Building Services: 3 of 4 quotes in the cohort went to replace. One skipped the leak search the same notes called for. | Future SP scorecard primitive. |
| Install-defect chains begin at install date, not at first failure. | Refuel 1328 fountain (asset 2152683): install 2026-04-17, first failure SR 2026-04-22 (5 days post-install). Three SRs in 6 days on the same asset. | `pre-dispatch-checks.md` (Check 2 recently-installed-asset trigger), `post-facto-application.md` (Check 5 post-implementation flag). |
| Warranty information not entered at install. | None of the 11 SRs had a warranty check in the review path. Two assets sampled (one installed 12 days ago) had zero warranty data. | `asset-data-lineage.md` (warranty status field treatment). Capture-at-install workflow is a separate primitive. |
| Callbacks are not analyzed as a category. | Refuel 1328 fountain: 3 SRs in 6 days on the same asset, only visible because Stephanie connected them by hand. | `sr-graph-resolver.md` (recurring-failure flag), `post-facto-application.md` (Check 4 recurring-failure flag). Callback metric primitive is a separate FacilitiesOS backlog item. |
| Multi-vendor installs need a different chargeback model. | Refuel 1328 fountain: Crete on mechanical, Air America on equipment commissioning. Current chargeback assumes one installer. | Captured as a known limitation in `pre-dispatch-checks.md` (Check 2). Multi-vendor model is a future workflow design item. |
| Action items get captured but disappear. | Stephanie's billing-side tracking holds, callback notes, OEM warranty handoff notes all get written somewhere then disappear. | This is exactly what `MAGIC/memory/actions.md` is for. The agent layer's job is to convert findings into tracked entries with owner and check-back. |
| Single-axis SR scoring is insufficient. | One axis cannot distinguish documentation failure (Air America) from diagnostic shortcut (Crete) from clean scope-boundary call (Fastcility). | Multi-axis scorecard proposal (cold beverage retro action #7) — separate work, heading to the SR-review leadership group. |

### Patterns surfaced and rule changes

| Pattern | Evidence (verified against VixxoLink) | Promoted to |
|---|---|---|
| Skill mis-pulls Facility Lead from notes / email cc traffic when the SR header `teamLeadFirstName` / `teamLeadLastName` field is correctly populated. The "in-SR Team Lead" field can also be manipulated by some teams as a routing workaround; the SR header field is the authoritative source. | Lacy Lakeview TX SR `SR-EXAMPLE-016`: SR header carries `teamLeadFirstName: Jainasia, teamLeadLastName: Vanholten`. Skill named Brandon Covington in field 7 because the notes (1-30FX5FQ, 1-30FN7MC, 1-30FZTAA) referenced Brandon as backup contact when Jainasia could not be reached. Brandon is the wrong FL for Texas; Jainasia owns it. | `pre-dispatch-checks.md` Check 10 (FL identification discipline). |
| Sibling SR detection by FWKD overlap is required when `vixxolink_get_related_service_requests` returns empty. Same FWKD on two SRs at the same site within a short window is a sibling signal, not two independent events in a cluster. | Lacy Lakeview TX SR `SR-EXAMPLE-016` (5/4) and `SR-EXAMPLE-017` (5/3) share `workOrderNumber: FWKD7498778`, same site `1-2N7C1DG`, same asset `ASSET-EXAMPLE-003`, same description, same issue. `get_related_service_requests` returned empty results on both; the sibling signal is in the FWKD field on the SR header. Skill counted them as two separate cluster members and quoted "fifth SR in 90 days" when the unique work-event count is four. | `sr-graph-resolver.md` Rule 6 (FWKD overlap as sibling signal). |
| 90-day same-site lookback misses same-asset recurring failures with prior repair history just outside the window. Hard-water failures on super-automatic espresso machines are the canonical case: a January boiler / solenoid / safety-valve replacement at the same asset is exactly the precedent the next failure needs. | Freeport ME asset `ASSET-EXAMPLE-004` SR `SR-EXAMPLE-018` created 2026-01-15 (109 days before 2026-05-04 run, 19 days outside the 90-day window). Customer NTE $2,491.00 lift approved by JVANHOLTEN on 1/29. CRS technician resolution note (rob@crsllcme.com, 2026-02-03): "Returned and installed parts ordered and approved... Location has very hard water." Visit note (ian@crsllcme.com, 1/22) named bad solenoid valve plus calcium buildup; parts list named in update note (RLIPAOPAO 2/3) matches Brandon's review verbatim. The skill missed this prior visit on the 5/4 Freeport SR triage and did not surface the hard-water signal that makes the next visit a category conversation, not a routine repair. | `pre-dispatch-checks.md` Check 1 / Check 9 (180-day lookback extension on same-asset cluster hits). |
| Cancelled SRs are not all alike. Some cancelled SRs have logged real work and should be counted in the cluster paired with their sibling. Others are billing-only clones of an already-completed SR, created with `Created for billing purposes only, do not dispatch` tracking holds, and should be suppressed to avoid double-counting one repair. | Lincoln ME SR `SR-EXAMPLE-019` created 2026-02-24 with description "Clone of FWKD6982897 - DO NOT DISPATCH." Tracking Hold note (BCOVINGTON 2026-02-24 19:38): "Created for billing purposes only, do not dispatch." Pine Tree confirmed cancellation same day. Original SR `SR-EXAMPLE-020` (created 2026-02-11) carries the actual repair, AR Successful, FWKD7065127. Including the cancelled clone in the asset cluster double-counts one repair. The framing in the team's review note ("we transferred the FWKD over to charge for incurred labor") is not what the data shows; the cancelled SR was never dispatched and never carried independent labor charges. | `sr-graph-resolver.md` Rule 7 (cancelled-SR cluster handling), `sub-status-decoder.md` (Cancelled sub-status with billing-clone subtype). Separately: the customer billing process that drove the clone is a customer-success conversation with Circle K AR, not a skill change. |
| Diagnosis lines that promote SP-stated symptoms to inferred root cause without explicit confidence and explicit signal create operator-trust loss. The right discipline is: quote the SP's literal words first, infer only with the volumetric or temporal signal that supports the inference, name confidence explicitly. | Lacy Lakeview TX SR `SR-EXAMPLE-017` Falcon 5 note (JMIRANDA 2026-05-04 05:05 UTC): "Tech diagnosed the issue and found out that they needed to add 40-60 pound of freon for 3 units. Freon - $95/pound, Labor - $250 / unit = $750. Customer Total w/Tax: $7,378.86." Skill packet promoted this to "Falcon 5 diagnosed refrigerant short across three units" and "system-level refrigerant leak now identified." Brandon's review correctly noted Falcon 5 did not say "leak" in the literal note. The skill should have framed the diagnosis with confidence and the volume signal: "SP states 40-60 lbs across 3 RTUs needed; HVAC standard practice and EPA 608 compliance treat refrigerant volume of this magnitude as leak signal; probable system leak, high confidence based on volume; recharge without leak detection is non-compliant scope and a near-certain callback." | `pre-dispatch-checks.md` Check 11 (diagnosis-vs-symptom provenance and confidence discipline). |
| Refrigerant volume per RTU is itself a diagnostic signal in HVAC standard practice. A request for 13-20 lbs of refrigerant per unit in a recharge scope is leak signal regardless of whether the SP uses the word "leak." Approving recharge without leak detection is non-compliant under EPA Section 608 / AIM Act enforcement and creates a near-certain callback. | Lacy Lakeview TX HVAC cluster: cumulative reactive R&M approaching $11,600 across 4 unique work events in 90 days (4, not 5, after FWKD-sibling dedupe of `SR-EXAMPLE-016` and `SR-EXAMPLE-017`); Falcon 5 quote $7,378.86 to recharge across 3 RTUs. Realistic leak-search-then-repair-then-recharge scope is $1,500 to $3,500. The reactive-vs-replace threshold for a c-store RTU set is in this range; engineering-assessment / RTU-replacement evaluation conversation is overdue. | `trades-hvac-r/reference/operations.md` (refrigerant-volume → leak-signal rule with EPA 608 framing). Cross-reference from `pre-dispatch-checks.md` Check 11. |
| Resolution-note quality check: a resolution note that says "opened wall" without saying "sealed wall" should fire a quality flag and block the SR from advancing to AR Successful. Same shape for any work that ends with an open structural condition. | Hooksett NH SR `SR-EXAMPLE-021` review (Brandon): tech reportedly opened the wall to access broken drain pipe; resolution notes say "Replaced drainage piping into wall, replaced shutoffs and mixing valve" but do not confirm the wall was sealed afterward. Need to verify on this specific SR but the rule shape is general. | Candidate for `quality-signals.md` (`repair-review` family, severity medium) once verified across a second site. Observed once today; appended here pending second occurrence per the file's promotion rule. |
| Specialized-trade SP-profile capability check: when a workorder requires a specialized certification (backflow testing, refrigerant handling, controlled substances, gas piping), the assigned SP should be verified against an SP-profile capability flag before dispatch. | Dover NH SR `SR-EXAMPLE-022` review (Brandon): backflow prevention test workorder routed to Marquis Contracting because Vixxo does not have a contractor formally loaded into Backflow coverage; SR went to "CK Review" rather than to a Backflow-certified SP. The SP profile capability flag (`BackflowTesting = Yes`) does not yet exist in the SP master and would need to be developed. | Future SP-onboarding-compliance primitive (FacilitiesOS backlog, ties to Derek's 2026-05-04 point 20). Captured here as observed; promote to a check once the SP-profile capability field exists in the data layer. |

## How to add to this file

When a new run surfaces a new pattern, add a row to the relevant table (Run A, Run B, Run C, or a new Run D/etc. section). Format per row: pattern (one sentence), evidence (specific SR or SRs with detail), promoted-to (which decoder rule, which check, or which separate operational action this becomes).

Promote a pattern from "observed once" to "decoder rule" only when:

1. The pattern repeats across runs OR is severe enough to warrant a one-shot rule.
2. The decoder rule has a clear input (what the agent sees), a clear output (what the agent should conclude), and a clear failure mode (what happens when the rule does not fire).
3. The rule does not require platform-side data the agent does not have today.

Patterns that fail any of these tests stay in this file as "observed" until either evidence accumulates or platform support arrives.
